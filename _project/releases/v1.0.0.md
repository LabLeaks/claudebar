# v1.0.0 (provisional)

## What "1.0" means

claudebar depends on undocumented Claude Code internals (.jsonl format, file layouts, CLI flags). When Claude Code ships a breaking change, claudebar breaks. 1.0 means we catch and fix regressions fast enough that users don't notice.

## Core: Regression Detection + Auto-Fix Pipeline

### Regression testing
- CI job runs on a schedule (daily or on Claude Code release)
- Tests exercise undocumented surfaces we depend on:
  - `.jsonl` transcript format and field names
  - `--resume`, `--settings`, `--teammate-mode` CLI behavior
  - Statusline JSON shape (model, rate_limits, context_window)
  - `~/.claude/projects/`, `~/.claude/tasks/`, `~/.claude/teams/` file layouts
  - Task list ID env var and task file format
- Tests run against real `claude` binary, not mocks
- Open question: do we need API calls (requires key in CI) or can we test CLI surface only?

### Auto-investigation
- When regression tests fail, a Claude agent:
  - Examines expected vs actual behavior
  - Checks Claude Code changelog/release notes if available
  - Classifies: mechanical fix (field rename, path move) vs structural change
  - Proposes a fix with explanation

### Auto-PR
- Mechanical fixes: cut a PR automatically for human review
- Structural changes: file an issue with analysis and options
- PR includes updated tests proving the fix works

### Auto-release
- Merged PR triggers GoReleaser + Homebrew formula update
- Every user gets the fix on next `claudebar` invocation (or sooner)

### Auto-update
- claudebar checks for new versions on startup (non-blocking)
- Updates itself without restarting the tmux session
  - Binary: rm + cp pattern (already proven in `make install`)
  - tmux config: re-source on existing server
  - Side panes: survive restart (already proven with respawn-pane)
- Zero-downtime: user's Claude conversation continues uninterrupted
